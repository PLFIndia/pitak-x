## 2. Secrets (Dart)

- **OK** ‚Äî `lib/core/crypto/secret_bytes.dart` ‚Äî Fully ¬ß6.1-compliant: mutable `Uint8List`, two-pass wipe (random then zero), use-after-dispose `StateError`, redacted `toString()`, `Random.secure()`. Honest GC-limitation documented.
- **OK** ‚Äî `lib/core/crypto/secure_passphrase_field.dart` ‚Äî Passphrase entry avoids full-String retention: growable wipeable byte buffer, per-keystroke UTF-8 encoding with delta-only transient String, mask-only TextField, old buffers wiped on grow, `takeSecret()` ownership transfer, autocorrect/suggestions disabled. Residual IME risk documented. Strong design.
- **OK** ‚Äî `lib/features/vault/application/vault_session_controller.dart` ‚Äî Held session secret wiped on lock (:292), on failed load (:436), replaced-with-dispose on rekey (:181) and re-hold (:441); `ref.onDispose` wipe registered in `build()` (:75). Clear ownership contract ("takes ownership of passphrase") on every entry point. Biometric S disposed after storing (:245). Fail-closed ordering on enrolment (store S before blob).
- **Minor** ‚Äî `lib/core/crypto/secure_passphrase_field.dart:184` ‚Äî On deletion the byte buffer is cleared but the masked `TextEditingController` retains its bullet text until re-rendered; bullets are non-secret so impact is nil ‚Äî noted only for completeness. No action needed.
- **Major** ‚Äî `lib/features/vault/infrastructure/ffi_vault_repository.dart:167,236,261-262,289,317,334,352,366,381,405` ‚Äî Every FFI call passes `passphrase.copyBytes()`, but the Dart-side copy is never wiped after the call. `copyBytes()` docs say "Caller owns the copy and must wipe it"; the comment at :164 claims "the Rust side zeroes its own copy" ‚Äî but flutter_rust_bridge marshals a *further* copy to Rust, and the Dart `Uint8List` handed to the binding lingers un-zeroed on the GC heap for every unlock and every CRUD write. Why it matters: multiplies un-wiped passphrase copies in memory, exactly what ¬ß6.1 tries to minimize. Direction: wrap each call in try/finally that zeroes the copy, or add `SecretBytes.useAsync` that wipes a temp copy after the awaited call.
- **Minor** ‚Äî `lib/features/vault/infrastructure/ffi_vault_repository.dart:290-296` ‚Äî Biometric secret S arrives from FFI as `wrap.secret` (plain `List<int>`); it is copied into `SecretBytes` but the original FFI-owned list is not wiped. Same GC-copy caveat; smaller surface (one-time enrolment). Direction: zero `wrap.secret` after copying.
- **Major** ‚Äî `lib/features/vault/infrastructure/secure_storage_biometric_keystore.dart:51,63` ‚Äî Biometric secret S transits as base64 `String` on both write (`secret.use(base64Encode)`) and read. The doc comment calls this "the unavoidable boundary cost" (plugin API is String-only) ‚Äî that's true, but the decoded `bytes` at :64 is copied into `SecretBytes` while the intermediate `base64Decode` result is not wiped, and the base64 String itself is a full un-wipeable encoding of S. ¬ß6.6 says avoid passing secrets through channels as String. Direction: at minimum wipe the intermediate list; longer-term consider a byte-capable secure-storage channel or store S wrapped so the String at rest isn't the raw secret.
- **OK** ‚Äî `secure_storage_biometric_keystore.dart` ‚Äî Hardened options are good: Android encryptedSharedPreferences, iOS `unlocked_this_device` (non-migrating). Errors carry no secret material.
- **OK** ‚Äî `ffi_vault_repository.dart` error mapping ‚Äî all FFI errors map to typed Failures; no secret/PII in diagnostics; NOT-NULL violations fail closed as corrupt.
- **OK** ‚Äî Ownership discipline in application layer: `restore_controller.dart:47-48`, `vault_controller.dart:46` dispose the passphrase in `finally`-style regardless of outcome; `open_vault_from_archive.dart` documents caller ownership. `vault_unlock_page.dart:44-48` disposes the `SecurePassphraseController` in widget dispose; failure path disposes the taken secret (:71-73).
- **OK** ‚Äî FLAG_SECURE (¬ß6.6): applied app-wide via `lib/main.dart:33-44` ‚Äî `ref.listen` on vault session state toggles `screenSecurityProvider.setSecure` whenever the vault is unlocked (PII visible). Narrow MethodChannel (`lib/core/platform/screen_security.dart`), no-op off Android. Passphrase-entry pages are covered whenever this fires; note it keys off vault state, not per-screen ‚Äî entry fields on an *uninitialized* vault (enable flow) may be capturable until unlock succeeds. Minor gap, worth a per-screen secure toggle on passphrase pages.
- **Minor** ‚Äî Background behavior (¬ß6.6): `lib/core/widgets/app_gate.dart:68-85` re-gates the UI on `paused`/`hidden`, fail-closed on unknown settings ‚Äî good. But the vault session (`keepAlive` provider holding `_passphrase`) is NOT locked/wiped on backgrounding; the secret persists in memory while the app sits in background. Deliberate UX trade-off (session survives), but ¬ß6.6 suggests clearing on `paused` ‚Äî worth an explicit timeout or documented decision.
- **OK** ‚Äî No `print`/`debugPrint`/logger calls emit secrets anywhere in `lib/` (searched, zero hits). No secret fields in JSON serialization paths reviewed.

## 3. Rust core

- **OK** ‚Äî `rust/src/crypto.rs` ‚Äî KDF: Argon2id v0x13, m=64 MiB, t=3, p=1, 32-byte output ‚Äî solid, OWASP-comparable params, matches the legacy Kotlin format. AEAD: AES-256-GCM via RustCrypto `aes-gcm`, fresh CSPRNG salt+IV per wrap (`OsRng`), no nonce reuse (new random IV every `wrap_vault_key` call; low volume makes random-nonce collision risk negligible). GCM tag verification is the constant-time authenticity check (library-provided); wrong-passphrase vs tampered ciphertext deliberately indistinguishable. KEK and unwrapped key both in `Zeroizing<>`.
- **OK** ‚Äî `rust/src/api.rs` ‚Äî Passphrase `Vec<u8>` taken by value and `zeroize()`d before every return on all entry points (unlock, all 6 writes via `with_vault_key`, create, rewrap ‚Äî both buffers ‚Äî and biometric wrap). Vault key/MK confined to `Zeroizing<>` inside Rust; only rows and the intentional biometric secret S cross FFI. Narrow typed command surface, no arbitrary-SQL/path command. All fallible paths return typed `Result` enums (no panic/unwrap on the hostile-input paths reviewed).
- **OK** ‚Äî `rust/src/vault.rs` ‚Äî All SQL parameterised; FK `ON DELETE RESTRICT` enforced via per-connection `PRAGMA foreign_keys=ON` on the write path (correctly documented SQLite footgun); wrong key detected by forced decrypt of `sqlite_master`; constraint vs not-found vs generic errors distinguished; `create_vault` refuses to clobber an existing file (fail-closed).
- **Minor** ‚Äî `rust/src/api.rs:60-64` (`unlock_and_read_vault` et al.) ‚Äî `zeroize()` on the passphrase wipes the Vec's final allocation, but `flutter_rust_bridge` deserialization and any Vec reallocation history mean earlier heap copies may survive. Inherent to the FFI marshalling layer, consistent with the documented best-effort posture; no action beyond documentation.
- **Minor** ‚Äî `rust/src/vault.rs:96-105,135-141` ‚Äî `sqlite3_key` return code checked, but on `rc != 0` or later failures the opened `Connection` is dropped without `sqlite3_rekey`/close ordering concerns ‚Äî fine ‚Äî however error strings embed `db_path` (`create_vault:91`) and raw SQLite error text. Paths aren't secrets here (app-owned dir), so Nit-level; keep an eye that no row data ever rides an error message.
- **Minor** ‚Äî `rust/src/api.rs` ‚Äî `db_path` accepted verbatim from Dart with no canonicalization/containment check; a compromised Dart side could point it anywhere writable. Given Flutter-side is the same trust domain (not a webview) and there's no arbitrary-SQL surface, risk is low, but a prefix check against the app-docs dir would harden the ¬ß2 "validate every IPC argument" posture.
- **OK** ‚Äî Test coverage of hostile/malformed paths is strong: `crypto.rs` unit tests (malformed blob, wrong passphrase, KAT vector for Argon2id, fresh salt/IV per wrap, non-32-byte key rejected); `vault.rs` unit tests (wrong key on write, NotFound, FK RESTRICT both directions, clobber refusal, round-trips, race-safe temp paths); `api.rs` unit tests (rewrap wrong-old-pass/malformed-blob, biometric wrong-secret/malformed-blob, old passphrase invalidated after rewrap, passphrase can't open bio blob); `rust/tests/vault_fixture.rs` hermetic CI gate through the exact FFI entrypoint with committed non-secret fixture, asserting Corrupt vs WrongPassphrase distinction; `rust/tests/real_archive.rs` env-gated acceptance test with no secrets in-repo.
- **Note** ‚Äî Temp files: the Rust core never creates temp files during vault open/save ‚Äî it operates directly on the caller-supplied `db_path` (SQLCipher may create `-wal`/`-shm` siblings, which inherit the DB's encryption). Archive extraction (where temp handling would matter) lives on the Dart side (`bounded_zip_extractor`), out of this scope. Tests clean up their own temp DBs including `-wal`/`-shm`.
- **Note** ‚Äî `lib/src/rust/api.dart` is FRB-generated (header confirms); it is a thin typed mirror of `api.rs` ‚Äî passphrases cross as `List<int>` (never String), errors as sealed freezed classes. Matches ¬ß6.6 "byte arrays over method channels". The un-wiped Dart-side copies are already flagged in ¬ß2 findings.
- **Nit** ‚Äî `rust/src/crypto.rs` ‚Äî AES-GCM uses empty AAD; binding the salt/IV (or a format version) as AAD would cheaply harden against mix-and-match blob-segment splicing. Format is legacy-locked to the Kotlin app, so only actionable on a future format bump.

## 4. Database & backup

- **OK** ‚Äî SQL injection: all runtime queries are parameterised. Drift codegen covers app queries; `legacy_db_reader.dart` uses constant SELECTs; `backup_archive_writer.dart` uses prepared statements with `?` placeholders for every row value. The only interpolations are compile-time constants (`_ftsColumns` in `app_database.dart`, Room `identity_hash` constants at `backup_archive_writer.dart:283-287`) ‚Äî not attacker-influenced.
- **OK** ‚Äî Hostile-archive handling: `bounded_zip_extractor.dart` is a credited Signal-inspired port with per-entry cap (200 MiB), total cap (500 MiB), entry-count cap (4096), zip-slip defence (leaf reduction + `/`, `\`, NUL rejection), duplicate/directory-entry/empty-name rejection, and verification of ACTUAL decompressed size (never trusting central-directory headers). Cover names re-validated via `CoverPaths.leafOf` at `restore_backup.dart:322` (defence in depth). Malformed rows: `legacy_db_reader.dart` coerces tolerantly (unknown enum tokens ‚Üí null, non-int ‚Üí null) rather than crashing.
- **OK** ‚Äî Fail-closed phase ordering in `restore_backup.dart`: everything fallible without side effects (extract, manifest parse, schema-too-new refusal, vault unlock, legacy reads) happens before the first device write; wrong-passphrase is distinct from corrupt; books+wishlist replaced inside a single Drift transaction (rollback preserves pre-restore rows on failure).
- **Major** ‚Äî `restore_backup.dart:186-218` + `vault_store.dart:106-116` ‚Äî Restore is not atomic ACROSS stores. After the Drift transaction commits (Phase 6), the vault install (Phase 6.5) can still fail ‚Üí device left with NEW library + OLD vault, reported as failure. Worse, inside `installRestored` the DB is renamed onto the live path BEFORE the blob is written (:113-114); a crash/IO failure between the two leaves a restored `borrowers.db` paired with the OLD wrapped-key blob ‚Üí vault permanently unopenable (wrong key). Why it matters: this is the one data-loss window in an otherwise careful pipeline. Direction: write blob to a temp file first, then rename DB and blob in a two-file commit (or write blob before the DB rename, since a new-blob/old-db mismatch is equally detectable but the blob is re-writable from the archive); consider staging both then renaming.
- **Minor** ‚Äî `restore_backup.dart:196-198` ‚Äî `rebuildFts()` runs after the transaction commits; if it throws, the code returns `StorageFailure` even though the library WAS replaced ‚Äî a misleading "failed" restore that actually applied. Direction: run rebuild inside the transaction or report partial success distinctly.
- **Minor** ‚Äî `restore_backup.dart:310-330` ‚Äî Phase 7 cover wipe/write is silent best-effort: wiping the covers dir can succeed and then individual writes fail, leaving books with dangling cover refs and no signal in `RestoreSummary`. Acceptable (placeholder renders), but a count of skipped covers in the summary would be honest.
- **Minor** ‚Äî `backup_archive_writer.dart:117` ‚Äî The live vault `borrowers.db` is copied byte-verbatim while the Rust core may have written it via short-lived connections. If SQLCipher ever runs in WAL mode, `-wal`/`-shm` siblings would be silently omitted ‚Üí stale/torn snapshot in the backup. Currently default (delete) journal mode makes this safe; worth an assertion or comment pinning the journal mode.
- **Minor** ‚Äî Whole-archive-in-RAM: extractor materialises up to 500 MiB decompressed plus the original bytes in memory (`restore_backup.dart` receives `archiveBytes` as one `Uint8List`; writer likewise builds the ZIP fully in memory). On low-end devices a legitimate large backup can OOM. Bounded and deliberate, but streaming to/from disk is the long-term fix.
- **OK** ‚Äî At-rest/location posture: work dirs (`backup_create_work`, `restore_work`) and covers live under the app documents dir (app-private on Android/iOS, not world-readable); scratch dirs are wiped in `finally` on both create and restore. Staged plaintext `books.db` (private notes/location fields) only ever exists inside the app sandbox transiently. Vault DB and blob are ciphertext at rest; backups leave the device only via the user-invoked OS share sheet. Books/wishlist are plaintext at rest by design (non-secret catalog per project threat model) ‚Äî consistent with the documented split.
- **OK** ‚Äî Migration path: no in-place Room‚ÜíDrift migration exists ‚Äî legacy DBs are read only through restore (one-time translation, `legacy_db_reader.dart` preserving `id` + `book_uid` so vault `loans.book_id` stays resolvable). Drift `schemaVersion` is 1 with `onCreate` only, so no forward/backward migration risk yet. Restore refuses `schemaVersion > knownSchemaVersion` (`backup_manifest.dart:28`, fail-closed) and tolerates unknown manifest fields (forward-compat booleans).
- **OK** ‚Äî Idempotency: restore is a wipe-and-replace on every store it touches; re-running the same archive converges to the same state.

## 6. Network & publish

- **OK** ‚Äî Token at rest: `secure_storage_publish_credential_store.dart` uses `flutter_secure_storage` with hardened options (Android `encryptedSharedPreferences: true`; iOS `unlocked_this_device`, non-migrating). Never SharedPreferences. Client id + repo kept in the same store deliberately.
- **OK** ‚Äî Token in transit: `http_github_api.dart:38-41` passes the token only as an `Authorization: Bearer` header, never in URLs or query params; hosts are hard-coded `https://github.com` / `https://api.github.com` (TLS enforced by scheme; no cert pinning ‚Äî acceptable for GitHub's public API, but ¬ß6.4 "where feasible" is unaddressed; worth a one-line justification comment).
- **Major** ‚Äî `http_github_api.dart:395-405` (`_excerpt`) + `publish_library_use_case.dart:280` ‚Äî HTTP error bodies (first 200 chars) flow verbatim into `PublishFailure('GitHub HTTP $code: $excerpt')`, which the UI renders. GitHub error JSON normally has no secrets, but a proxy/captive portal or unexpected response could inject arbitrary content into the visible message, and `_guard`'s `on Exception catch (e) => GitHubApiException('$e')` stringifies arbitrary exceptions (could embed the full request URL). No token can appear (it's header-only), but this violates the "safe message only" convention. Direction: map status codes to fixed messages; keep the excerpt in a debug-only detail.
- **Minor** ‚Äî `http_github_api.dart` ‚Äî no handling of 401/403 rate-limit or token-revoked responses distinct from other errors: no retry/backoff anywhere (single-shot per call), no `Retry-After`/`X-RateLimit-Remaining` inspection. Single-user publish volume makes limits unlikely, but a revoked token surfaces as an opaque "GitHub HTTP 401" instead of "sign in again". Direction: special-case 401 ‚Üí typed auth failure prompting re-auth.
- **Minor** ‚Äî `http_github_api.dart:  pollAccessToken` ‚Äî the device-flow `slow_down` handling exists in `github_device_flow.dart` (+5s), matching the RFC 8628 minimum; fine. But `requestDeviceCode`/`pollAccessToken` send the client_id as a form body over TLS ‚Äî correct, nothing leaks.
- **Minor** ‚Äî Token as `String` throughout (`token()` ‚Üí `String?`, threaded through every API call). Same ¬ß6.1 exception as noted in ¬ß5 for `DeviceFlowSuccess`: HTTP header APIs force String, so it's borderline-unavoidable, but the token also sits in the credential-store Future results and use-case locals with GC-managed lifetime. Document the accepted trade-off in the store's doc comment.
- **OK** ‚Äî Data minimization (what leaves the device): `publish_library_use_case.dart` publishes only redacted books (`redactForPublish`, F-01), coarse availability ("available"/"out", omitted when vault locked ‚Äî no borrower/loan data ever), salted cover paths (no internal ids), library name + explicitly opt-in contact fields. Commit message is `Pitaka publish <epoch>` ‚Äî no PII.
- **OK** ‚Äî Atomicity: Git Data flow moves the branch ref exactly once at the end (`commitFiles` step 5, `force: false`); a failed publish leaves the live page untouched. Manifest saved only after commit success.
- **OK** ‚Äî No telemetry: no analytics/crash-reporting/beacon calls anywhere in the publish path; every outbound call is user-invoked.
- **Major** ‚Äî `lib/features/publish/presentation/pages/publish_page.dart:118-127` ‚Äî in `_signIn`, the `await for` loop awaits `_showUserCodeDialog(...)` inside the stream body. Async-generator backpressure means the device-flow generator is SUSPENDED at its `yield` while the dialog is open ‚Äî polling for the token does not run until the user taps "Done", despite the dialog saying "Waiting for you to authorize‚Ä¶ this dialog can stay open." If the user authorizes in the browser and leaves the dialog open, sign-in never completes. Direction: fire the dialog without awaiting it (unawaited + dismiss on success), or consume the stream via `listen`.
- **Minor** ‚Äî `publish_page.dart:139,253,481` ‚Äî raw error strings reach the UI: `'Sign-in failed: $reason'` (DeviceFlowFailed carries `GitHubApiException.message`, which via `_guard` can be a stringified arbitrary exception) and `'Could not list repos: ${e.message}'`. Compounds the ¬ß6 `_excerpt` finding ‚Äî same direction: fixed messages.
- **OK** ‚Äî `publish_page.dart` token hygiene: token is written straight from `DeviceFlowSuccess` into the secure store and never held in widget state, displayed, or logged; sign-out clears it; user code (not a secret) is the only thing copied to clipboard; page states the privacy contract ("Borrower data ‚Ä¶ never included") matching what the use case actually does. Basic-info tab publishes only user-typed, explicitly-public fields.
- **Minor** ‚Äî `lib/features/lookup/infrastructure/open_library_lookup_service.dart:46` & `google_books_lookup_service.dart:73` ‚Äî no request timeout on lookup GETs (unlike `BoundedCoverFetcher`, which has one). A hung connection stalls the add-book lookup indefinitely with no typed failure. Direction: `.timeout(...)` mapped to `LookupNetworkError`.
- **Minor** ‚Äî Both lookup services: `on Exception catch (e) => LookupNetworkError('$e')` embeds raw exception text in the result's message. Harmless if the UI shows a generic "lookup failed", but verify the consumer; prefer a constant message.
- **Needs verification** ‚Äî Google Books `imageLinks.thumbnail` is historically an `http://` URL. If stored as `coverUrl` and later fetched by the publish path, TLS depends entirely on `CoverUrlAllowList`/`BoundedCoverFetcher` enforcing https + host allow-list (M1 comment claims it does; that file was outside this scope). Confirm the allow-list rejects/upgrades plain http.
- **OK** ‚Äî Lookup data minimization: outbound query params carry only the ISBN or user-typed title (public bibliographic data) to two hard-coded https hosts; lookups run only on explicit user action (per providers.dart doc); results cached in-memory per session only. No telemetry.

## Summary

Top issues by severity:

1. **Major (¬ß6)** ‚Äî Device-flow sign-in stalls: `publish_page.dart:118-127` awaits the user-code dialog inside `await for`, suspending token polling until the dialog closes ‚Äî GitHub sign-in likely never completes if the user follows the on-screen instructions.
2. **Major (¬ß5)** ‚Äî Swallowed write failures: `library_controller.dart:72,79` and `wishlist_controller.dart:45` discard `Either` results from remove/restore/delete ‚Äî failures silently look like success (violates fail-closed).
3. **Major (¬ß6)** ‚Äî Raw HTTP bodies/exception strings surface to users: `http_github_api.dart` `_excerpt`/`_guard` ‚Üí `PublishFailure`/`_status` messages (`publish_page.dart:139,253`) ‚Äî replace with fixed, typed messages.
4. **Major (¬ß7)** ‚Äî Business logic + direct repository IO in widgets: cover capture pipeline in `book_detail_page.dart:318` and logo picker in `settings_page.dart:463` bypass the application layer, are untestable and (logo) silently swallow errors.
5. **Major (¬ß5)** ‚Äî Library sort is read non-reactively (`library_controller.dart:91` `ref.read` of settings in the load path), so changing the sort setting doesn't re-sort the list until an unrelated refresh.

Overall: architecture discipline is high (clean layering, exemplary vault secret lifecycle, minimized publish payload, zero telemetry, strong security-test culture). The recurring weaknesses are a handful of swallowed `Either` failures, raw error text leaking to the UI, and two widget-embedded IO pipelines that should be controllers.
